# Bundle proposals — Coastal Brewing Co. (2026-05-09)

Owner directive 2026-05-09: I propose, owner ratifies. Bundle math
anchored to `pricing-margin-model-2026-05-09.md` retail prices.

Each bundle includes:
- **Components** (what's in it)
- **Sum of retail** (if bought separately)
- **Bundle price** (what we charge)
- **Bundle savings to custee**
- **Coastal margin** (our actual COGS vs bundle revenue)
- **Pairing rationale** (why these go together)
- **Sourcing readiness** (any items not yet stocked)

Margin floor: every bundle must net ≥ 25% Coastal margin (after COGS +
fulfillment) at the bundle price. Bundles below that floor get
restructured.

| **Custee saves** | $59.97 - $67.97 (full retail) | |
| **Coastal margin** | recouped over 3 months of sub @ ~$9/month gross | ✅ payback at month 3 |

**Logic**: This is a CAC investment, not a margin play. Sub LTV math:
- Month 1: cost basis $28-29 (mug + coaster + coffee + card + ship)
- Months 2-12: sub revenue $21.24/month × 11 = $233.64

- Month-12 Coastal net: $233.64 − ($5.50 × 11) − $28.28 month-1 cost = **$144.86 over year-1** at single-cup-tier

**Sourcing**: Coffee = TCR. Mug + coaster = Coastal merch program.

## Sourcing readiness gate

Before any bundle ships, the bundle's components must ALL be:

1. In Coastal catalog (`scripts/catalog.py`).
2. Inventory available (drop-ship items: TCR; landed items: in Pooler
   warehouse).
3. Stripe Product+Price record minted in Stripe Dashboard for the
   bundle SKU.
4. Photographed for storefront listing (Iller_Ang).
5. Tested via internal order (Coastal staff orders to validate flow).

## Stripe Checkout encoding

Each bundle gets a Stripe Product with line-item bundle SKU
(e.g. `BUNDLE-ETHIOPIAN-CEREMONY`). When custee triggers a bundle
purchase from chat or storefront, the Checkout Session is created with
that SKU as a single line-item priced at the bundle price. Bundle
contents go into Stripe metadata for fulfillment-side splitting (TCR
gets the coffee SKU; Coastal warehouse picks the accessory SKUs).
